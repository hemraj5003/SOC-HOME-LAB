# SOC-HOME-LAB
Introduction

Introduction
For understanding how real attacks happen and how defenders detect and evade them, I started designing my own SOC home lab. My goal was simple: understand how the architecture works, how attackers try to gain access, and how systems are configured and monitored in a real security environment.

When I started learning cybersecurity, I had many questions. Instead of only watching tutorials, I decided to build a lab environment myself. While doing this, I realized that building things practically creates even more questions, but it also helps in understanding concepts much deeper.

At first, I started setting up the lab on my own laptop. Due to storage limitations, I had to carefully manage where each virtual machine would run. I planned to run Kali Linux as the attacker machine and Ubuntu as the security monitoring server.

However, my first attempt did not go smoothly. Because of incorrect configuration on my Ubuntu system, the setup got stuck and my machine started behaving unexpectedly. I spent almost two hours troubleshooting the problem, but during that process I learned something important — how Wazuh installation and configuration actually works.

That small failure helped me understand the system much better.

Once I fixed the configuration issues, I knew what kind of setup I wanted, but I was still unsure about one important question:

What should the architecture of my SOC home lab actually look like?

Designing the SOC Home Lab

To answer that question, I started researching how real organizations design their security monitoring environments.

I watched tutorials, read documentation, and studied how SIEM platforms collect logs from endpoints and analyze them for suspicious activity.

Most examples showed a similar architecture:

A security monitoring server

A victim system generating logs

An attacker system simulating malicious activity

Based on this idea, I designed a simple architecture for my lab.

Kali Linux (Attacker)  -->  Attack Simulation

Windows Host (Wazuh Agent)  -->  Security Logs

Ubuntu Server (Wazuh Manager)  -->  Log Analysis

Wazuh Dashboard

In this setup:

Kali Linux
Acts as the attacker machine used to simulate attacks.

Windows
Acts as the monitored endpoint with the Wazuh agent installed.

Ubuntu Server
Runs the Wazuh manager and dashboard, which collects and analyzes logs from the Windows system.

Network Configuration

To make the environment work properly, I used different network modes for each machine.

Ubuntu Server (Wazuh Manager)

Network Mode: NAT

Purpose: Security monitoring and log analysis server.

Windows Machine (Victim System)

Network Mode: Bridged Adapter

Purpose: Generate system logs and security events.

Kali Linux (Attacker Machine)

Network Mode: Bridged Adapter

Purpose: Perform attack simulations against the Windows machine.

This configuration allows Kali and Windows to communicate directly, while Ubuntu collects and analyzes the logs generated by the Windows agent.

Wazuh Installation

Running Wazuh requires a system with sufficient resources. For this lab environment, at least 6 GB RAM is recommended to ensure the Wazuh manager and dashboard run smoothly.

The installation was done on the Ubuntu server using the official Wazuh installation script.

Download the installation script:

curl -sO https://packages.wazuh.com/4.14/wazuh-install.sh

Run the installation:

sudo bash wazuh-install.sh -a

The installation script automatically installs:

Wazuh Manager

Wazuh Indexer

Wazuh Dashboard

After the installation finished, the Wazuh dashboard became accessible through the web browser, allowing monitoring and investigation of security alerts.

Attack Simulation in the Lab

Once the lab environment was ready, I started testing simple attack techniques from Kali Linux to observe how logs were generated.

Some of the basic activities I tested include:

Network Scanning

Using Nmap, I performed network reconnaissance to discover open ports on the Windows machine.

nmap -sS <target-ip>

This simulates the reconnaissance phase of an attacker.

SMB Enumeration

Attackers often explore shared resources on Windows systems.

Initially, I could not see the SMB shares because the Windows firewall was blocking the requests. After allowing the required access from Windows PowerShell, the shares became visible during scanning.

Once the firewall rules were adjusted, I could see the SMB shares from Kali.

SMB Login Attempts

I tried connecting to the SMB shares using smbclient.

First, I tested fake credentials, which resulted in authentication failures. These failed attempts were visible in the Wazuh dashboard as unauthenticated login attempts.

Later, when I used the correct username and password, I was able to access the shared resources and could see the Administrator C drive and D drive.

However, the connection was closed almost immediately, which helped me observe how such access attempts and disconnections appear in security logs.

DoS Simulation

I also experimented with basic packet flooding using hping3 to simulate abnormal traffic patterns.

hping3 --flood -S <target-ip>

This helped observe how unusual network activity can generate alerts in the monitoring system.

Monitoring with Wazuh

The most interesting part of this lab was observing how security logs are generated and analyzed.

The workflow looks like this:

Kali performs an attack simulation

Windows generates security logs

The Wazuh agent sends logs to the Ubuntu server

Wazuh analyzes the logs using detection rules

Alerts appear in the Wazuh dashboard

This process helped me understand how Security Operations Centers (SOCs) monitor systems and investigate suspicious behavior.
